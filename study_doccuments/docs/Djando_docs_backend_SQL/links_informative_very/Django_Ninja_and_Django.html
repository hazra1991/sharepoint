🧭 Overview: NinjaAPI() vs Router()

ComponentPurpose
NinjaAPI()The main API application object (like the root API handler). It connects to Django’s URL system.
Router()A modular sub-router that lets you group endpoints logically.

🎯 When to use what?

  • ✅ Use NinjaAPI() when you’re defining the root API or very simple APIs.

  • ✅ Use Router() when your project has multiple apps or many endpoints — it helps keep your code modular and maintainable.

🔧 Example 1: Simple API (no router)

python
# project/api.py
from ninja import NinjaAPI

api = NinjaAPI()

@api.get("/hello")
def say_hello(request):
    return {"message": "Hello world"}

✅ Result:

  • Your API is available at /api/hello if you wire api.urls to /api/ in urls.py.

🔧 Example 2: Using Router() for modular structure

Let's say you have an app called bookmarks with its own endpoints.

python
# bookmarks/api.py
from ninja import Router

router = Router()

@router.get("/list")
def list_bookmarks(request):
    return [{"title": "Django", "url": "https://www.djangoproject.com"}]

Then in your main API setup:

python
# project/api.py
from ninja import NinjaAPI
from bookmarks.api import router as bookmarks_router

api = NinjaAPI()

# Mount router under a path
api.add_router("/bookmarks/", bookmarks_router)

✅ Result:

  • Your API endpoint is available at /api/bookmarks/list.

This structure keeps things clean and organized, especially as your app grows.

✅ True — and here's why:

🧠 Concept: One NinjaAPI() per project, many Router()s

🔷 api = NinjaAPI()

  • You should only define this once in a project.

  • It acts as the central hub, similar to how Django uses urlpatterns as the single entry point to URL routing.

🔷 router = Router()

  • You can create as many routers as you want, often one per Django app (e.g., users, blog, products).

  • Then you register them to the main API using api.add_router(...).

📦 Analogy

Think of it like a train station:

  • NinjaAPI() = the main train station (handles all traffic).

  • Each Router() = a platform or gate with its own destinations.

You're telling Django:

“Route all API traffic through this central API object, and here's how to handle different groups of endpoints.”

✅ Example Project Structure

plaintext
project/
├── users/
│   └── api.py      ← router = Router()
├── bookmarks/
│   └── api.py      ← router = Router()
├── project/
│   └── api.py      ← api = NinjaAPI(); add all routers
│   └── urls.py     ← connects api.urls to /api/

project/api.py

from users.api import router as users_router
from bookmarks.api import router as bookmarks_router

api.add_router("/bookmarks/", bookmarks_router)

project/urls.py

python
from django.urls import path
from project.api import api

urlpatterns = [
    path("api/", api.urls),  # one central entry point
]

🧠 Why only one NinjaAPI()?

  • Django routes HTTP requests to one main URL dispatcher.

  • NinjaAPI wraps this dispatcher for API endpoints.

  • If you define more than one NinjaAPI, it would be confusing for Django — you'd have to manually mount and separate each one.

  • It's cleaner and idiomatic to keep a single API object and use routers for separation.

Can I define multiple NinjaAPI() instances in a Django project and register them in urlpatterns like this?

Short Answer: ✅ Yes, you can do it technically, but ⛔ it's almost always unnecessary, uncommon, and can cause confusion or mismanagement.

🔍 Let's walk through it in detail

✅ Your code is valid Django + Ninja syntax:

python

from project.api import api, api2

    path("api/", api.urls),
    path("api2/", api2.urls),
]

This will NOT throw an error. Django allows you to mount as many URL handlers as you want — even multiple NinjaAPI() instances.

🎯 But here’s what happens (and why you usually avoid this):

🤖 What NinjaAPI() does:

Each NinjaAPI():

  • Creates its own set of views, schemas, and exception handling.

  • Registers its own OpenAPI docs (/docs), authentication, versioning, etc.

  • Is totally independent.

So when you do this:

python
api = NinjaAPI()
api2 = NinjaAPI()

You are creating two separate APIs. Each has:

  • Its own /docs page

  • Its own routes

  • No shared routers unless you mount them manually in both

🧪 Example:

project/api1.py

python
from ninja import NinjaAPI
api = NinjaAPI()

@api.get("/hello")
def hello(request):
    return {"msg": "Hello from API 1"}

project/api2.py

python
from ninja import NinjaAPI
api2 = NinjaAPI()

@api2.get("/world")
def world(request):
    return {"msg": "Hello from API 2"}

project/urls.py

python
from django.urls import path
from project.api1 import api
from project.api2 import api2

urlpatterns = [
    path("api/", api.urls),
    path("api2/", api2.urls),
]

Now you can access:

  • http://localhost:8000/api/hello

  • http://localhost:8000/api/docs

  • http://localhost:8000/api2/world

  • http://localhost:8000/api2/docs

And that’s totally valid.

😕 But... Why is this usually a bad idea?

🔻 1. Confusing structure

You now have to maintain two separate docs pages, two separate routers, and two sets of behaviors.

🔻 2. Duplicated logic

If you want shared things (auth, exceptions, schema overrides), you’d need to configure them in both APIs.

🔻 3. Breaks modularity

The design pattern Django Ninja promotes is:

✅ One central API app → organize with routers per app

Not:

❌ One API object per app → each app controls its own NinjaAPI

✅ Correct & Scalable Pattern (Best Practice)

python
# project/api.py
from ninja import NinjaAPI
from users.api import router as users_router
from blog.api import router as blog_router

api = NinjaAPI()
api.add_router("/users/", users_router)
api.add_router("/blog/", blog_router)

Then in urls.py:

python
from django.urls import path
from project.api import api

urlpatterns = [
    path("api/", api.urls),
]
