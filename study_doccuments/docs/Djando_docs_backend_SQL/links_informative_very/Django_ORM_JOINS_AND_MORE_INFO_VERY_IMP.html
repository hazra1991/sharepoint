Yes, you can perform joins on unrelated models in Django — but not using the ORM's __ relationship traversal.

🔴 1. You CANNOT use __ if models aren't related

Example:

python
# This will raise error if not related
Book.objects.filter(car__brand='Toyota')  # ❌

Why? Because Django has no way to know how to join Book to Car — unless there’s a ForeignKey/OneToOne path.

✅ 2. Option: Use Raw SQL Join (low-level)

python
from django.db import connection

raw_query = '''
    SELECT b.*
    FROM myapp_book b
    JOIN myapp_car c ON b.title = c.model_name
    WHERE c.brand = %s
'''

Book.objects.raw(raw_query, ['Toyota'])

  • ✅ Can join any tables

  • ❌ No queryset chaining

  • ❌ No ORM filtering

Use this only when needed — it’s powerful but bypasses Django’s safety.

✅ 3. Option: Use Subqueries with OuterRef and Exists (Modern Django ORM style)

Let's say Book and Car aren't related, but they both have a field like name, and you want to find books that match cars.

python
from django.db.models import OuterRef, Subquery, Exists
from myapp.models import Book, Car

cars_qs = Car.objects.filter(model_name=OuterRef('title'))

Book.objects.annotate(
    has_matching_car=Exists(cars_qs)
).filter(has_matching_car=True)

✅ This is a JOIN-like filter without a direct relation
✅ Runs efficiently as a correlated subquery in SQL
✅ 100% ORM-safe and composable

✅ 4. Option: .annotate() with Subquery

Example:

python
from django.db.models import Subquery, OuterRef

matching_brand = Car.objects.filter(model_name=OuterRef('title')).values('brand')[:1]

Book.objects.annotate(car_brand=Subquery(matching_brand)).filter(car_brand='Toyota')

This mimics a JOIN by pulling in fields from another table based on custom logic.
