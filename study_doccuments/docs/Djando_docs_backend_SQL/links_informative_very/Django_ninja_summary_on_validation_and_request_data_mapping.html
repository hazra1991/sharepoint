1) If you name a query parameter request (same as the injected HTTP request), how does Django Ninja distinguish them?

Core concept:

  • Django Ninja injects the HTTP request object only if your function has a parameter named exactly request and its type is NOT one of the recognized request data types (like int, str, pydantic.BaseModel, etc.).

  • Its a special positional parameter convention at the very start (or anywhere), but Django Ninja expects only one request parameter to be the actual HTTP request.

What if you have a query param named request?

Example:

python
@api.get("/items")
def get_items(request: HttpRequest, request: str):
    ...

This is invalid Python because you cannot have two parameters with the same name.

What if you do:

python
@api.get("/items")
def get_items(request, request_param: str):
    ...

  • Here, request (no type hint) is the injected HTTP request by naming convention.

  • request_param is a query parameter extracted from the URL query string, named request_param.

What if you want to access query parameter named request?

You cannot name your function parameter request for this because it conflicts with the injected request object.

Instead:

python
from ninja import Query

@api.get("/items")
def get_items(request, request_: str = Query(..., alias="request")):
    # `request` is Django HttpRequest
    # `request_` is the query parameter named "request"
    return {"http_request_method": request.method, "request_param": request_}

  • Here, request is the Django HttpRequest.

  • request_ is the function parameter name (to avoid clash).

  • alias="request" tells Ninja to look for the query parameter named "request" in the URL.

So, how does Django Ninja know which is which?

  • By parameter names:

    • If the function param is named request inject the Django HttpRequest object (unless annotated otherwise).

    • For any other name, treat it as a query/form/path param or whatever type you declare.

  • By type hints:

    • If you annotate the parameter with a Pydantic model, Query, Form, Path, etc., Django Ninja will parse from the correct place.

  • The naming conflict is resolved by you: You cannot have two parameters named request. If you want the HTTP request and a query parameter named "request", you must differentiate the names in the function and use alias to match the query parameter.

What content types does Django Ninja support by default for body parsing?

  • JSON only (default):

    • Django Ninja internally calls request.json() or request.body and uses Pydantic's JSON parsing.

    • The client must send:

      http
      Content-Type: application/json

    • Request body JSON is validated and parsed into the UserSchema object.

  • Other content types (like form data or multipart):

    • For form data, you can use Form parameters explicitly.

    • Example:

      python
      def create_user(request, name: str = Form(...), age: int = Form(...)):
    ...

  • XML:

    • Not supported by default.

    • Django Ninja has no built-in XML parsing support.

    • To accept XML, you must:

      • Implement custom parsing middleware or request body processing.

      • Parse XML manually inside the view (e.g., request.body parse XML validate).

So, in your example, only JSON request bodies will be parsed and validated automatically.

Can you add support for other formats?

  • Yes, but requires extra work:

    • Write a custom parser.

    • Use third-party libraries to parse XML or other formats.

    • Override the request processing or add a custom decorator.

  • By default, Django Ninja assumes request body is JSON for Pydantic models.

2. Request Injection The request Parameter

How it works:

  • If a parameter is named request, Django Ninja injects the raw Django HttpRequest object.

  • No type annotation is required, but optional: request: HttpRequest.

Example:

python
@api.get("/status")
def status(request):
    return {"method": request.method}

  • If you want a query parameter named request, avoid naming conflicts:

python
from ninja import Query

@api.get("/check")
def check_request(request, request_param: str = Query(..., alias="request")):
    return {"req": request_param}

3. Query Parameters (GET / URL params)

Implicit way (auto-mapped by name and type):

python
@api.get("/search")

    return {"keyword": keyword, "page": page}

Explicit way (with Query()):

python
from ninja import Query

@api.get("/search")
def search(
    keyword: str = Query(..., description="Search term"),
    page: int = Query(1, ge=1)
):
    return {"keyword": keyword, "page": page}

Aliasing:

python
@api.get("/data")
def get_data(val: str = Query(..., alias="value1"), va: str = Query(..., alias="value2")):
