Behind the scenes: What Django does

  • The dev server (runserver) runs using:

    bash
    python manage.py runserver

    which internally uses:

    python
    WSGIServer(..., ThreadingMixIn)

  • That means: Each HTTP request is handled in a new thread (via Python's threading module).

  • So, blocking functions like time.sleep() do not block the whole server, only the current thread.

Production Servers (Gunicorn, Daphne, etc.)

In production, you typically run Django behind a WSGI or ASGI server:

Detailed Breakdown

1. What is WSGI?

WSGI = Web Server Gateway Interface

  • The original Python web server standard.

  • Django runs on WSGI by default.

  • Threaded concurrency: each request is handled in a thread or process.

  • Blocking operations (e.g., time.sleep, DB access) block the thread.

You write:

python
def view(request):
    # normal sync code
    return JsonResponse(...)

  • Works great for most use cases.

  • Used with Gunicorn, uWSGI, etc.

2. What is ASGI?

ASGI = Asynchronous Server Gateway Interface

  • Newer standard (Django 3.0+ supports it).

  • Designed for async views, WebSockets, and event-driven concurrency.

  • Lets you write non-blocking async def views.

Example:

python
import asyncio
from django.http import JsonResponse

async def async_view(request):
    await asyncio.sleep(5)  # non-blocking wait
    return JsonResponse({"msg": "Async Response"})

  • Used with Uvicorn, Daphne, etc.

3. Do you need to rewrite views?

If your view isWorks under WSGI?Works under ASGI?
def view(request): Yes Yes
async def view(request): No (wont run) Yes

So: under ASGI you can use both. Under WSGI, only def-style sync views are valid.

4. Why use ASGI if WSGI can handle concurrency too?

Great question the key differences lie in how concurrency is achieved and what types of workloads you have.

ComparisonWSGIASGI
Uses threads/processes Yes Yes
Handles async views No Yes
Scalable for I/O-bound tasks (e.g., DB, APIs) Threads block Non-blocking with await
Handles WebSockets No Yes
Performance with 10k idle clients Bad (threads scale poorly) Great (event loop scales well)

Real-World Guidance

Use CaseRecommended
Standard APIs, no WebSockets WSGI or ASGI
Want WebSockets / server push ASGI
Want to use async def views ASGI
Heavy CPU tasks (e.g., image processing) WSGI (thread/process model is fine)
Heavy I/O tasks (e.g., many external API calls) ASGI (non-blocking = better performance)

Django's Current Position

  • Django supports both WSGI and ASGI out of the box.

  • In asgi.py Django projects use:

    python
    application = get_asgi_application()

  • Your code doesn't break but you can go async if you want to.

1. Is your assumption correct?

WSGI or ASGI both use/spin a different thread/process per request hit?
And the asyncio event loop is handled inside the view if needed?

Almost correct with small refinements:

WSGI:

  • WSGI is sync-only.

  • Each request is handled by a thread or a process (depending on the WSGI server: Gunicorn, uWSGI, etc.)

  • Django handles the request in that thread one thread per request.

  • Blocking view code (e.g., time.sleep()) blocks that thread.

ASGI:

  • ASGI supports sync and async.

  • Each request is scheduled on an asyncio event loop (usually inside Uvicorn or Daphne).

  • Async views (async def) are run in the event loop.

  • Sync views (def) are run in a thread pool (via asgiref.sync.sync_to_async) to avoid blocking the event loop.

So:

  • ASGI doesn't create a thread per async request instead, it schedules async views inside a single-threaded event loop.

  • But sync views still run on threads (in a pool).

2. Who spawns threads/processes?

The app server (Gunicorn, Uvicorn, Daphne, etc.) is responsible for all concurrency.

ServerWho spins threads or processesNotes
Gunicorn (WSGI) Gunicorn creates workers (processes or threads)Django is passive just handles request when called
uWSGI (WSGI) Same as GunicornHighly configurable concurrency
Uvicorn (ASGI) Uvicorn runs event loop, thread poolAsync views use loop, sync views use threads
Django's runserver (WSGI) Dev server uses ThreadingMixInSingle process, multi-threaded

Django does NOT manage threads or concurrency its purely the app servers job.

3. What does the application server (Gunicorn/Uvicorn) actually do?

Think of the server as the bridge between HTTP requests and your Django app.

Here's what an app server does:

  1. Accepts raw HTTP connections (socket-level)

  2. Spins up workers (processes or threads or an event loop)

  3. Calls into your Django application (via application = get_wsgi_application() or get_asgi_application())

  4. Handles request lifecycle: reads request, passes it in, gets response, writes it back to the socket

  5. Manages concurrency: scaling, process/thread lifecycle, timeouts

4. Where does the async event loop live?

In an ASGI server like Uvicorn, the event loop is:

  • Created once when the server starts.

  • Managed entirely by Uvicorn.

  • Your Django app registers as a coroutine callable, like:

python
application = get_asgi_application()

  • When Uvicorn receives an HTTP request:

    • If the Django view is async def: runs it in the event loop.

    • If the view is def: offloads it to a thread pool to avoid blocking.
