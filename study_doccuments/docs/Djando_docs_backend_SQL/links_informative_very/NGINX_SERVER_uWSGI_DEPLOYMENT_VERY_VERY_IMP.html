Great questions! Let me explain both parts clearly:

1️⃣ What is cpetools_api_autodeploy.sock?

What it is:

  • It's a Unix domain socket file.

  • Think of it as a special "file" that allows fast inter-process communication (IPC) between Nginx and uWSGI on the same machine.

  • Instead of using a TCP/IP port (like 127.0.0.1:8000), uWSGI creates this socket file and listens for requests there.

  • Nginx sends requests into this socket, and uWSGI reads them.

What’s inside it?

  • Not a text file — you can’t open or read it like a normal file.

  • It carries the raw uWSGI protocol data between Nginx and uWSGI.

  • Contains the HTTP request info encoded in the uWSGI protocol, which is a lightweight binary protocol optimized for this use case.

Why Unix socket?

  • Faster than TCP because it avoids network stack overhead.

  • More secure — only local processes can connect.

  • Often used in production for performance and security reasons.

2️⃣ What does include uwsgi_params do?

Context:

  • Nginx uses the uwsgi_pass directive to send requests to uWSGI.

  • But uWSGI expects a very specific protocol with certain environment variables representing the HTTP request.

  • uwsgi_params is a file containing a list of these variables and how to pass them from Nginx to uWSGI.

What is inside uwsgi_params?

Typical example /etc/nginx/uwsgi_params content (might vary slightly):

nginx
uwsgi_param  QUERY_STRING       $query_string;
uwsgi_param  REQUEST_METHOD     $request_method;
uwsgi_param  CONTENT_TYPE       $content_type;
uwsgi_param  CONTENT_LENGTH     $content_length;

uwsgi_param  SCRIPT_NAME        "";
uwsgi_param  REQUEST_URI        $request_uri;
uwsgi_param  PATH_INFO          $document_uri;
uwsgi_param  DOCUMENT_ROOT      $document_root;
uwsgi_param  SERVER_PROTOCOL    $server_protocol;
uwsgi_param  REQUEST_SCHEME     $scheme;
uwsgi_param  HTTPS              $https if_not_empty;

uwsgi_param  REMOTE_ADDR        $remote_addr;
uwsgi_param  REMOTE_PORT        $remote_port;
uwsgi_param  SERVER_PORT        $server_port;
uwsgi_param  SERVER_NAME        $server_name;

What does this mean?

  • It tells Nginx how to package the HTTP request info (method, URI, headers, client IP, etc.) into the uWSGI protocol.

  • When uWSGI receives the request, it uses these variables as if they were environment variables.

  • Django then accesses these values to build request.META and handle the HTTP request normally.

Is the HTTP body lost?

  • No! The HTTP request body (POST data, JSON, form data) is passed as a raw stream over the same socket.

  • uwsgi_params only sets environment-like variables — the body goes through automatically.

  • So, your Django app still gets the full HTTP request: headers, query params, and body.

What if you write your own WSGI server or use something else?

  • The uWSGI protocol is NOT a universal standard like HTTP; it is specific to uWSGI and Nginx’s module for uWSGI.

  • If you write your own WSGI server (say a custom one), Nginx won’t automatically know how to talk to it via uwsgi_pass unless it supports the uWSGI protocol.

  • You have a few options:

    1. Use standard protocols that Nginx knows:

    2. Implement the uWSGI protocol in your server (hard and uncommon).

    3. Use other protocols Nginx supports, for example:

      • fastcgi_pass (used for PHP-FPM)

      • proxy_pass (generic HTTP proxy)
